**Spec:** `docs/superpowers/specs/2026-06-13-integration-suite-flow.md`
(canonical contract; this plan is the executable form).

**Goal.** Add an optional `flow:` ordering expression to
`tools/integration-suite-multisite/` that orchestrates the existing
`input:` (fires) and `expected:` (observations) by reference. After
this plan ships:
- A scenario without `flow:` runs exactly as it does today (multi-
  input executor, flat expected — every existing scenario untouched).
- A scenario with `flow:` routes to the flow executor: observations
  positioned between fires become gates; tail parallel-observe groups
  cover end-state checks in one window; per-id verdicts in the report.

**Scope discipline.** Implement exactly what the spec describes —
nothing else. Specifically OUT (with future-spec homes per spec §11):
parallel **fires** (the bracket grammar reserves the shape, the
loader rejects with "deferred to concurrency slice"), named
orderings, predicate-based observation indexing, chaos. If a task
wants to "while we're here…" — stop.

- TDD, every task. Tests RED before implementation; never the inverse.
- Existing 23+ scenarios stay green at every commit. The validator
  output must match the recon baseline (Task 0) verbatim after every
