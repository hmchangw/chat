# E2E Test Suite — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up a two-site backend e2e test suite driven by `testcontainers-go`'s compose module against a new `docker-local/e2e/compose.e2e.yaml`. Tests exercise the full request path through real NATS subjects and HTTP routes — no mocks, no service code modifications — across two independent sites with full federation wiring (NATS supercluster via gateways).

**Architecture:** New `e2e/` Go package at the repo root, build-tagged `//go:build e2e`. Stack lifecycle is owned by `TestMain` via `tc.NewDockerCompose(...).Up(ctx)`. Federation wiring (cross-site `INBOX_*` `Sources + SubjectTransforms`) is a harness-owned function called after compose-up — matching the prod-reference shape from `2026-04-27-inbox-stream-ownership-design`. Test isolation is `siteID`-per-test, no global teardown between tests.

**Tech Stack:** Go 1.25, `testcontainers-go` (compose module), Docker, Docker Compose v2 / v1.29+, NATS supercluster (gateways), MongoDB v8, Cassandra 5, Elasticsearch 8.17, Valkey 8, Keycloak 26.

**Source spec:** `docs/superpowers/specs/2026-05-08-e2e-tests-design.md`. The spec describes the architecture; this plan executes it chapter by chapter.

**Prerequisite:** `pkg/testutil/keycloak/` already landed (commit `ba9af4c`).

**Out of scope:**
- Frontend (`chat-frontend`) — backend boundary only.
- Modifying existing service code or per-service Dockerfiles / composes.
- A `.github/workflows/e2e.yml` — designed-in via the `make e2e` contract, not built here.
- Worker `/healthz` endpoints — harness probes worker readiness via NATS `consumer info` polling instead.

## Chapter 1 — Scaffolding & bootstrap

**Goal:** Establish the directory `docker-local/e2e/`, the checked-in `.env.example` with every image-ref env var defaulted, the gitignore for the auto-created `.env`, and the Makefile targets that drive the suite. Nothing in this chapter starts containers — it's pure scaffolding so subsequent chapters have a stable home for compose + configs.

After this chapter, `make e2e-up` will fail (no compose file yet) but `make e2e-up`'s prerequisite — auto-creating `.env` from `.env.example` — works and is idempotent.

### Task 1.3 — Add Makefile targets

**Files:** `Makefile` (modify)

- [ ] **Step 1: Read the existing Makefile** to find the right insertion point. Append the new section after the existing test/lint targets so e2e sits at the end as an opt-in suite.

- [ ] **Step 2: Append the e2e block.**

```makefile
# ---------------------------------------------------------------------------
# E2E suite. Two-site backend stack via testcontainers + docker compose.
# See docs/superpowers/plans/2026-05-08-e2e-tests.md.
# ---------------------------------------------------------------------------
E2E_DIR     := docker-local/e2e
E2E_COMPOSE := $(E2E_DIR)/compose.e2e.yaml
E2E_ENV     := $(E2E_DIR)/.env

# Auto-create .env from .env.example on first run; idempotent.
$(E2E_ENV):
	@cp $(E2E_DIR)/.env.example $@
	@echo "Created $@ from .env.example."
	@echo "Edit it to point at an internal registry mirror, or leave defaults for upstream."

.PHONY: e2e e2e-only e2e-up e2e-down e2e-logs

# Full run: TestMain owns stack lifecycle.
e2e: $(E2E_ENV)
	go test -tags e2e -race -count=1 ./e2e/...

# Iteration loop: assumes a stack already running via `make e2e-up`.
e2e-only:
	E2E_REUSE_STACK=1 go test -tags e2e -race -count=1 ./e2e/...

# Manual stack control.
e2e-up: $(E2E_ENV)
	docker compose -f $(E2E_COMPOSE) up -d --wait

e2e-down:
	docker compose -f $(E2E_COMPOSE) down -v

# `make e2e-logs SERVICE=msg-gk-a` to tail one container.
e2e-logs:
	docker compose -f $(E2E_COMPOSE) logs -f $(SERVICE)
```
