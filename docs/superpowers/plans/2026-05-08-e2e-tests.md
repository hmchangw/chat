# E2E Test Suite — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up a two-site backend e2e test suite driven by `testcontainers-go`'s compose module against a new `docker-local/e2e/compose.e2e.yaml`. Tests exercise the full request path through real NATS subjects and HTTP routes — no mocks, no service code modifications — across two independent sites with full federation wiring (NATS supercluster via gateways).

**Architecture:** New `e2e/` Go package at the repo root, build-tagged `//go:build e2e`. Stack lifecycle is owned by `TestMain` via `tc.NewDockerCompose(...).Up(ctx)`. Federation wiring (cross-site `INBOX_*` `Sources + SubjectTransforms`) is a harness-owned function called after compose-up — matching the prod-reference shape from `2026-04-27-inbox-stream-ownership-design`. Test isolation is `siteID`-per-test, no global teardown between tests.

**Tech Stack:** Go 1.25, `testcontainers-go` (compose module), Docker, Docker Compose v2 / v1.29+, NATS supercluster (gateways), MongoDB v8, Cassandra 5, Elasticsearch 8.17, Valkey 8, Keycloak 26.

**Source spec:** `docs/superpowers/specs/2026-05-08-e2e-tests-design.md`. The spec describes the architecture; this plan executes it chapter by chapter.

**Prerequisite:** `pkg/testutil/keycloak/` already landed (commit `ba9af4c`).

**Out of scope:**
- Frontend (`chat-frontend`) — backend boundary only.
- Modifying existing service code or per-service Dockerfiles / composes.
- A `.github/workflows/e2e.yml` — designed-in via the `make e2e` contract, not built here.
- Worker `/healthz` endpoints — harness probes worker readiness via NATS `consumer info` polling instead.

## Chapter 2 — NATS configs (gateway-based supercluster)

**Goal:** Two NATS server configurations (`nats-a.conf`, `nats-b.conf`) that form a NATS supercluster via gateways. Each site is its own cluster with its own JetStream domain; gateways carry cross-site traffic so JetStream `Sources` from chapter 9 work without external API prefixes. Pattern is taken directly from `broadcast-worker/deploy/test/nats/nats-site{1,2}.conf` — proven to work in the existing two-site test fixture.

After this chapter, the configs exist on disk and can be hand-mounted into a NATS container for verification, but are not yet referenced by any compose file. Chapter 3 wires them into compose.

### Background — why gateways, not leaf nodes

The repo's existing two-site test config uses gateways (see `broadcast-worker/deploy/test/nats/`). Gateways are NATS' canonical pattern for federating independent clusters into a supercluster: each cluster keeps its own client trust domain and JetStream metadata, and gateway connections carry interest-based subject propagation between them. Cross-cluster JetStream `Sources` work transparently — the `Sources[].Name` references a stream by name, and the gateway resolves it on the remote cluster without needing `External.APIPrefix` configuration on the source side.

Leaf nodes are a different pattern (one cluster as a downstream of another, asymmetric trust relationship) that doesn't match what we need here.

### Task 2.1 — `nats-a.conf`

**Files:** `docker-local/e2e/nats/nats-a.conf` (new)

- [ ] **Step 1: Write the config.**

```hocon
# docker-local/e2e/nats/nats-a.conf
# Site A NATS server. Forms a supercluster with site B via gateway on port 7222.

server_name: nats-a
listen:      0.0.0.0:4222
http_port:   8222

jetstream {
  store_dir: /data/jetstream
  domain:    siteA
}

cluster {
  name:   siteA
  listen: 0.0.0.0:6222
  routes: []
}

gateway {
  name:    siteA
  listen:  0.0.0.0:7222
  gateways: [
    { name: siteB, urls: ["nats://nats-b:7222"] }
  ]
}
```

Notes that matter:
- `jetstream.domain: siteA` makes JetStream's per-cluster domain explicit. Without it, supercluster JetStream sourcing can ambiguate stream names between clusters. With it, `INBOX_siteA` lives in domain `siteA` and `OUTBOX_siteB` lives in domain `siteB`, and gateway-mediated `Sources` resolve unambiguously.
- `cluster.routes: []` is a single-node cluster — the e2e topology doesn't need clustered NATS within a site. The block is still present so the cluster has a name (gateways require a named cluster).
- `gateway.gateways[].urls` references `nats-b` by docker-network alias. The compose file in chapter 3 must use that exact service name for site B's NATS.
- `store_dir: /data/jetstream` matches the volume mount pattern from `docker-local/compose.deps.yaml`. Compose file in chapter 3 mounts a named volume there.

## Chapter 3 — Compose: site A dependencies

**Goal:** Create `docker-local/e2e/compose.e2e.yaml` populated with the six site-A dependency containers (nats-a, mongo-a, cass-a, es-a, valkey-a, keycloak-a). Every image ref uses `${E2E_*_IMAGE:-default}` so a fresh clone runs unchanged and the in-house port edits `.env` to override. All container names use the `-a` suffix to leave room for the parallel site B set in chapter 5. Healthchecks gate on protocol-level readiness, not just process up.

After this chapter, `make e2e-up` brings up all six site-A deps and they pass healthchecks; nothing else (services, site B) exists yet so e2e-up exits successfully with just the deps running.

### Conventions for the entire compose file

Locked in here, applied identically in chapters 4 and 5:
- **Compose-spec style**: no `version:` header, top-level `name: chat-e2e`, `depends_on:` with `condition: service_healthy` per `docker-local/compose.deps.yaml`. Compatible with both `docker-compose` v1.29+ and `docker compose` v2.
- **Network**: a single `chat-e2e` user-defined bridge that every service joins. Site A and site B share the network — gateway peering needs DNS resolution across sites.
- **Volumes**: named volumes per stateful container (`mongo-a-data`, `cass-a-data`, etc.) so `make e2e-down -v` cleans state.
- **Image refs**: `image: ${E2E_<DEP>_IMAGE:-<default>}` for every dep. The default in compose duplicates `.env.example` so the file works even if `.env` is missing.
- **Resource caps**: NONE. Defaults everywhere. Matches the spec's commitment to a 32GB / 8-core target.
- **Container names**: explicit `container_name:` set to the docker-network alias (`nats-a`, `mongo-a`, etc.) so logs in `make e2e-logs` match the alias and so other services can reference them by name.

### Task 3.2 — `nats-a` service block

- [ ] **Step 1: Append the nats-a block under `services:`.**

```yaml
  nats-a:
    image: ${E2E_NATS_IMAGE:-nats:2.11-alpine}
    container_name: nats-a
    networks: [chat-e2e]
    ports:
      - "4222:4222"   # client (host-exposed for harness + manual debugging)
      - "8222:8222"   # http monitoring
    volumes:
      - ./nats/nats-a.conf:/etc/nats/nats-server.conf:ro
      - nats-a-data:/data/jetstream
    command: ["-c", "/etc/nats/nats-server.conf"]
    healthcheck:
      test: ["CMD-SHELL", "wget -q -O- http://localhost:8222/healthz | grep -q ok"]
      interval: 2s
      timeout: 2s
      retries: 30
      start_period: 5s
```

Notes:
- The healthcheck hits `/healthz` on the monitoring port, which returns `200 OK` only after JetStream is ready. This is stronger than `nats-server`'s default port-open check.
- `wget` ships in `nats:*-alpine`. If the in-house port mirrors a different base, the healthcheck command may need adjustment.
- Port 7222 (gateway) is intentionally NOT host-mapped — only `nats-b` needs to reach it, over the docker network.

### Task 3.4 — `cass-a` service block

- [ ] **Step 1: Append.**

```yaml
  cass-a:
    image: ${E2E_CASS_IMAGE:-cassandra:5}
    container_name: cass-a
    networks: [chat-e2e]
    ports:
      - "9042:9042"
    volumes:
      - cass-a-data:/var/lib/cassandra
      - ../cassandra/init:/docker-entrypoint-initdb.d:ro
    environment:
      CASSANDRA_CLUSTER_NAME: chat-e2e-a
      CASSANDRA_DC: dc-a
      CASSANDRA_ENDPOINT_SNITCH: GossipingPropertyFileSnitch
    healthcheck:
      test: ["CMD-SHELL", "cqlsh -e 'describe keyspaces' >/dev/null 2>&1"]
      interval: 5s
      timeout: 5s
      retries: 60
      start_period: 30s
```

Notes:
- `../cassandra/init:/docker-entrypoint-initdb.d:ro` mounts the existing 10-file schema directory (`docker-local/cassandra/init/`) into Cassandra's init hook, which executes `*.cql` files in lexical order on first start. Same mechanism used by `docker-local/compose.deps.yaml`.
- Healthcheck `start_period: 30s` is generous because Cassandra cold-start on a fresh volume runs the schema init scripts before becoming queryable.
- `retries: 60` × `interval: 5s` = 5-minute health budget, also generous — first cold start with a fresh volume can take 60–90s before `cqlsh` accepts queries.

## Chapter 4 — NATS auth bootstrap + Compose site A services

**Goal:** Wire up the eleven site-A service containers and the NATS operator/account JWT bootstrap they all depend on. The existing `docker-local/setup.sh` generates a single-site NATS auth chain (operator + chatapp account + backend.creds + signing key); the e2e suite needs an analogous bootstrap producing artifacts under `docker-local/e2e/secrets/` that **both** sites' NATS share.

After this chapter, `make e2e-up` brings the full site-A stack online (deps + services) with operator-authed NATS, services bootstrapping their streams, and the auth-service ↔ Keycloak ↔ NATS-JWT-signing chain wired correctly.

### Background — NATS auth model the existing repo uses

`docker-local/setup.sh` produces:
- `operator.jwt` — root of trust for the local NATS deployment
- `account.jwt` (chatapp) + account seed — the user-facing namespace
- `sys.jwt` — system account for monitoring/health
- `backend.creds` — a creds file every backend service mounts at `/etc/nats/backend.creds` and connects with via `NATS_CREDS_FILE`
- `nats.conf` — embeds operator + `resolver_preload` of the two account JWTs
- `docker-local/.env` — exports `AUTH_SIGNING_KEY` (the chatapp account seed) so `auth-service` can sign user JWTs

**For a NATS supercluster the simplest correct choice is to share one operator + one chatapp account across both sites.** Gateways federate accounts across clusters; using the same operator/account JWTs in both `resolver_preload` blocks means a JWT issued by `auth-service-a` is accepted by `nats-b` and vice-versa, which matches how prod federation actually behaves. (Two operators with cross-signing is the more general pattern but unnecessary for the test environment.)

This chapter therefore:
1. Adds `docker-local/e2e/setup-e2e.sh` that generates ONE operator + ONE chatapp account + ONE SYS account + ONE backend.creds, plus an `auth.conf` snippet to `$include` from each site's NATS config.
2. Amends the chapter-2 NATS configs to `$include` the auth snippet.
3. Updates the chapter-1 Makefile to invoke `setup-e2e.sh` as a prerequisite to `e2e-up` (idempotent — skips if artifacts already exist).
4. Adds eleven site-A service blocks to `compose.e2e.yaml`.

### Task 4.1 — `setup-e2e.sh` generates NATS auth artifacts

**Files:** `docker-local/e2e/setup-e2e.sh` (new, executable)

- [ ] **Step 1: Read the existing `docker-local/setup.sh` end-to-end.** This task adapts it; understand it before forking.

- [ ] **Step 2: Write the e2e variant.** Key differences from `setup.sh`:
  - Outputs go to `docker-local/e2e/secrets/` (gitignored), not `docker-local/`.
  - Produces `auth.conf` fragment containing `operator: …` + `resolver: MEMORY` + `resolver_preload { … }`. NATS configs `$include` it.
  - Produces `e2e-secrets.env` fragment exporting `AUTH_SIGNING_KEY` for the `auth-service-a` and `auth-service-b` containers (compose `env_file` directive).
  - Idempotent: if `secrets/operator.jwt` exists, exit 0 silently (re-running `make e2e-up` doesn't regenerate keys).

```bash
#!/usr/bin/env bash
# docker-local/e2e/setup-e2e.sh
#
# Generates NATS operator + chatapp account JWTs + backend.creds for the e2e
# supercluster. Both site A and site B NATS instances use the SAME operator
# and chatapp account so JWTs are accepted by either cluster (gateway-federated
# accounts). Idempotent: skips if artifacts already exist.
#
# Outputs:
#   secrets/operator.jwt
#   secrets/account.jwt
#   secrets/sys.jwt
#   secrets/backend.creds
#   secrets/auth.conf       # $include'd from both nats-{a,b}.conf
#   secrets/e2e-secrets.env # AUTH_SIGNING_KEY for auth-service containers

set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
SECRETS_DIR="$SCRIPT_DIR/secrets"

if [ -f "$SECRETS_DIR/operator.jwt" ]; then
    exit 0   # already generated
fi

mkdir -p "$SECRETS_DIR"

NATS_BOX_IMAGE="${NATS_BOX_IMAGE:-natsio/nats-box:latest}"
TMPDIR=$(mktemp -d)
trap 'rm -rf "$TMPDIR"' EXIT

docker run --rm -v "$TMPDIR:/output" "$NATS_BOX_IMAGE" sh -c '
    set -e
    nsc add operator --name e2e --sys
    nsc env -o e2e >/dev/null
    nsc add account --name chatapp
    nsc edit account chatapp --js-mem-storage 512M --js-disk-storage 5G --js-streams 32
    nsc add user --account chatapp --name backend
    nsc describe operator --raw > /output/operator.jwt
    nsc describe account chatapp --raw > /output/account.jwt
    nsc describe account SYS --raw > /output/sys.jwt
    nsc generate creds --account chatapp --name backend > /output/backend.creds
    nsc describe account chatapp --field nats.signing_keys.0 --raw > /output/account.pub
    nsc env --field nkey-store-dir 2>&1 | grep -oE "/[^[:space:]]+" | head -1 > /output/store.txt
    cat $(cat /output/store.txt)/keys/keys/A/*/A*.nk > /output/account.seed
    nsc describe operator --field sub --raw > /output/operator.pub
    nsc describe account SYS --field sub --raw > /output/sys.pub
'

cp "$TMPDIR/operator.jwt"  "$SECRETS_DIR/operator.jwt"
cp "$TMPDIR/account.jwt"   "$SECRETS_DIR/account.jwt"
cp "$TMPDIR/sys.jwt"       "$SECRETS_DIR/sys.jwt"
cp "$TMPDIR/backend.creds" "$SECRETS_DIR/backend.creds"

OPERATOR_JWT=$(cat "$TMPDIR/operator.jwt")
ACCOUNT_JWT=$(cat "$TMPDIR/account.jwt")
SYS_JWT=$(cat "$TMPDIR/sys.jwt")
ACCOUNT_PUB=$(cat "$TMPDIR/account.pub")
SYS_PUB=$(cat "$TMPDIR/sys.pub")
ACCOUNT_SEED=$(cat "$TMPDIR/account.seed")

cat > "$SECRETS_DIR/auth.conf" <<EOF
operator: ${OPERATOR_JWT}
resolver: MEMORY
resolver_preload {
  ${ACCOUNT_PUB}: ${ACCOUNT_JWT}
  ${SYS_PUB}: ${SYS_JWT}
}
EOF

cat > "$SECRETS_DIR/e2e-secrets.env" <<EOF
AUTH_SIGNING_KEY=${ACCOUNT_SEED}
EOF
chmod 600 "$SECRETS_DIR/e2e-secrets.env" "$SECRETS_DIR/backend.creds"

echo "[setup-e2e] wrote secrets to $SECRETS_DIR"
```

## Chapter 7 — `harness/stack.go`: compose lifecycle

**Goal:** Implement the stack lifecycle in `e2e/harness/stack.go` using `testcontainers-go`'s compose module. Wire `TestMain` so it brings the stack up before the first test, tears it down after the last, and supports `E2E_REUSE_STACK=1` for fast iteration.

After this chapter, `make e2e` runs `go test`, brings up the compose stack, idles (no tests yet — chapter 10 adds the first), and tears it down. `make e2e-only` against an externally-running stack returns immediately.

### Background — `testcontainers-go/modules/compose`

The compose module's main entry point is `tc.NewDockerCompose(paths ...string)` returning a `ComposeStack`. Calling `.Up(ctx, opts...)` brings up the stack via the embedded compose-v2 engine (no shelling out), waits for healthchecks (`tc.WithRecreate(api.RecreateNever)`, `compose.Wait(true)`), and returns. `.Down(ctx, opts...)` tears down with optional volume removal. `ServiceContainer(ctx, name)` returns a `*testcontainers.DockerContainer` for a specific compose service so the harness can fetch host-mapped ports if needed.

The compose module's `WithReuse` option marks containers so re-runs against the same compose project don't recreate them — the natural fit for `E2E_REUSE_STACK=1`.

```go
//go:build e2e

package harness

import (
	"context"
	"errors"
	"fmt"
	"os"
	"path/filepath"
	"runtime"

	tcompose "github.com/testcontainers/testcontainers-go/modules/compose"
)

// Stack holds the running e2e topology: the compose project plus per-site
// endpoint URLs the test code can use to talk to NATS, Mongo, Cassandra,
// Elasticsearch, Valkey, Keycloak, and auth-service over the host network.
type Stack struct {
	compose tcompose.ComposeStack
	reused  bool

	SiteA SiteEndpoints
	SiteB SiteEndpoints
}

// SiteEndpoints holds host-network URLs for one site's exposed deps + auth.
// All values are populated regardless of E2E_REUSE_STACK because they're
// derived from the docker-compose port mappings, not from runtime inspection.
type SiteEndpoints struct {
	SiteID         string
	NATSURL        string
	NATSCredsFile  string // host path to backend.creds
	MongoURI       string
	CassandraHosts []string
	ESURL          string
	ValkeyAddr     string
	KeycloakURL    string
	AuthURL        string
}

// Start brings up the e2e topology and waits for healthchecks, OR (when
// E2E_REUSE_STACK=1) attaches to an already-running stack. Caller is
// responsible for calling Stop.
func Start(ctx context.Context) (*Stack, error) {
	composePath, err := composeFilePath()
	if err != nil {
		return nil, fmt.Errorf("locate compose file: %w", err)
	}

	stack := &Stack{
		SiteA: siteAEndpoints(filepath.Dir(composePath)),
		SiteB: siteBEndpoints(filepath.Dir(composePath)),
	}

	if reuse, _ := strconv.ParseBool(os.Getenv("E2E_REUSE_STACK")); reuse {
		stack.reused = true
		// Caller assumes the stack is running; harness verifies endpoints
		// during client construction (chapter 8).
		return stack, nil
	}

	cs, err := tcompose.NewDockerCompose(composePath)
	if err != nil {
		return nil, fmt.Errorf("create compose stack: %w", err)
	}
	stack.compose = cs

	if err := cs.Up(ctx, tcompose.Wait(true)); err != nil {
		_ = cs.Down(ctx, tcompose.RemoveOrphans(true), tcompose.RemoveVolumes(true))
		return nil, fmt.Errorf("compose up: %w", err)
	}

	return stack, nil
}

// Stop tears the stack down (volumes too). No-op when the stack was reused.
func (s *Stack) Stop(ctx context.Context) error {
	if s.reused || s.compose == nil {
		return nil
	}
	return s.compose.Down(ctx, tcompose.RemoveOrphans(true), tcompose.RemoveVolumes(true))
}

// composeFilePath finds docker-local/e2e/compose.e2e.yaml by walking up from
// this source file to the repo root. Avoids depending on the test process's
// CWD, which differs between `go test ./...` and IDE invocations.
func composeFilePath() (string, error) {
	_, thisFile, _, ok := runtime.Caller(0)
	if !ok {
		return "", errors.New("runtime.Caller unavailable")
	}
	// e2e/harness/stack.go → repo root: ../../
	root := filepath.Join(filepath.Dir(thisFile), "..", "..")
	path := filepath.Join(root, "docker-local", "e2e", "compose.e2e.yaml")
	if _, err := os.Stat(path); err != nil {
		return "", fmt.Errorf("compose file %s: %w", path, err)
	}
	return path, nil
}

// siteAEndpoints / siteBEndpoints return the host-network URL set the test
// code uses to talk to each site. Hard-coded against the host port mappings
// declared in compose.e2e.yaml (chapters 3 + 5). composeDir is the absolute
// path to docker-local/e2e/, used to compute the host path to backend.creds.
func siteAEndpoints(composeDir string) SiteEndpoints {
	return SiteEndpoints{
		SiteID:         "siteA",
		NATSURL:        "nats://localhost:4222",
		NATSCredsFile:  filepath.Join(composeDir, "secrets", "backend.creds"),
		MongoURI:       "mongodb://localhost:27017",
		CassandraHosts: []string{"localhost:9042"},
		ESURL:          "http://localhost:9200",
		ValkeyAddr:     "localhost:6379",
		KeycloakURL:    "http://localhost:8180",
		AuthURL:        "http://localhost:8080",
	}
}

func siteBEndpoints(composeDir string) SiteEndpoints {
	return SiteEndpoints{
		SiteID:         "siteB",
		NATSURL:        "nats://localhost:4322",
		NATSCredsFile:  filepath.Join(composeDir, "secrets", "backend.creds"),
		MongoURI:       "mongodb://localhost:27018",
```

```go
	"fmt"

// stack is the package-level singleton. Tests reach it via the helper
// accessors in test files (Stack(t), SiteA(t), SiteB(t)).
var stack *harness.Stack

func TestMain(m *testing.M) {
	os.Exit(runTestMain(m))
}

func runTestMain(m *testing.M) int {
	ctx, cancel := context.WithTimeout(context.Background(), 10*time.Minute)
	defer cancel()

	s, err := harness.Start(ctx)
	if err != nil {
		fmt.Fprintf(os.Stderr, "e2e: harness.Start: %v\n", err)
		return 1
	}
	stack = s

	code := m.Run()

	stopCtx, stopCancel := context.WithTimeout(context.Background(), 2*time.Minute)
	defer stopCancel()
	if err := s.Stop(stopCtx); err != nil {
		fmt.Fprintf(os.Stderr, "e2e: stack.Stop: %v\n", err)
		// Don't override a passing-test exit code with a teardown error;
		// log and continue.
	}
	return code
}

// Stack returns the running stack handle. Test files call this to reach
// per-site clients via stack.SiteA / stack.SiteB.
func Stack() *harness.Stack { return stack }
```

```go
package harness

import (
	"context"
	"fmt"
	"testing"

	"github.com/nats-io/nats.go"
	"github.com/nats-io/nkeys"
	"github.com/stretchr/testify/require"

	"github.com/hmchangw/chat/pkg/testutil/keycloak"
)

// Identity bundles everything a test needs to act as a specific user on a
// specific site: the realm-level account name, an authenticated NATS conn
// scoped to the user's JWT permissions, and the public NKey backing it.
type Identity struct {
	Account string
	NKey    nkeys.KeyPair
	NATSJWT string
	conn    *nats.Conn
}

// Conn returns the user-authenticated NATS connection.
func (id *Identity) Conn() *nats.Conn { return id.conn }

// Authenticate walks the Keycloak → auth-service → NATS chain for the given
// realm user. The realm uses fixed test users (alice, bob, etc.) preprovisioned
// in auth-service/deploy/keycloak/realm-export.json with password=password.
func (s SiteEndpoints) Authenticate(t *testing.T, ctx context.Context, account string) *Identity {
	t.Helper()

	kc := keycloak.NewClient(s.KeycloakURL, "chatapp")
	ssoToken, err := kc.AccessToken(ctx, "nats-chat", account, "password")
	require.NoError(t, err, "keycloak password grant for %s on %s", account, s.SiteID)

	kp, err := nkeys.CreateUser()
	require.NoError(t, err)
	pub, err := kp.PublicKey()
	require.NoError(t, err)

	authClient := s.HTTPClient(t)
	var resp struct {
		NATSJWT string `json:"natsJwt"`
	}
	httpResp, err := authClient.R().
		SetContext(ctx).
		SetBody(map[string]string{
			"ssoToken":      ssoToken,
			"natsPublicKey": pub,
			"account":       account,
		}).
		SetResult(&resp).
		Post("/auth")
	require.NoError(t, err, "POST /auth on %s", s.SiteID)
	require.False(t, httpResp.IsError(), "auth-service rejected: %d %s", httpResp.StatusCode(), httpResp.String())

	conn, err := nats.Connect(s.NATSURL,
		nats.UserJWTAndSeed(resp.NATSJWT, mustSeed(t, kp)),
	)
	require.NoError(t, err, "user-jwt nats connect on %s", s.SiteID)
	t.Cleanup(conn.Close)

	return &Identity{
		Account: account,
		NKey:    kp,
		NATSJWT: resp.NATSJWT,
		conn:    conn,
	}
}

func mustSeed(t *testing.T, kp nkeys.KeyPair) string {
	t.Helper()
	seed, err := kp.Seed()
	require.NoError(t, err)
	return string(seed)
}
```

```go
//go:build e2e

package harness

import (
	"context"
	"fmt"
	"time"

	"github.com/nats-io/nats.go/jetstream"

	"github.com/hmchangw/chat/pkg/stream"
)

// BootstrapFederation layers cross-site Sources onto each site's INBOX_*
// stream so events published to one site's OUTBOX flow into the peer's INBOX
// with the canonical chat.inbox.{site}.aggregate.> destination subject.
//
// Mirrors the prod-reference config from
// docs/superpowers/specs/2026-04-27-inbox-stream-ownership-design.md
// exactly (same Source name, same SubjectTransform shape).
//
// Idempotent against re-runs: UpdateStream rewrites the same config every
// time. Inbox streams must already exist (created by inbox-worker via
// BOOTSTRAP_STREAMS=true at container startup).
func BootstrapFederation(ctx context.Context, stack *Stack) error {
	if err := updateInbox(ctx, stack.SiteA, stack.SiteB.SiteID); err != nil {
		return fmt.Errorf("federation siteA<-siteB: %w", err)
	}
	if err := updateInbox(ctx, stack.SiteB, stack.SiteA.SiteID); err != nil {
		return fmt.Errorf("federation siteB<-siteA: %w", err)
	}
	return nil
}

// updateInbox attaches a Source to siteEndpoints.INBOX that pulls from
// `OUTBOX_<peerSiteID>` (which lives on the OTHER cluster, federated via
// the supercluster gateway).
func updateInbox(ctx context.Context, site SiteEndpoints, peerSiteID string) error {
	conn, err := nats.Connect(site.NATSURL, nats.UserCredentials(site.NATSCredsFile))
	if err != nil {
		return fmt.Errorf("connect %s: %w", site.NATSURL, err)
	}
	defer conn.Close()

	js, err := jetstream.New(conn)
	if err != nil {
		return fmt.Errorf("jetstream context: %w", err)
	}

	cfg := stream.Inbox(site.SiteID)
	src := fmt.Sprintf("outbox.%s.to.%s.>", peerSiteID, site.SiteID)
	dst := fmt.Sprintf("chat.inbox.%s.aggregate.>", site.SiteID)

	// Wait until inbox-worker has created the stream (BOOTSTRAP_STREAMS path
	// runs at service startup; healthcheck-gated dep ordering can't gate on
	// this because workers don't have healthchecks).
	if err := waitForStream(ctx, js, cfg.Name); err != nil {
		return err
	}

	_, err = js.UpdateStream(ctx, jetstream.StreamConfig{
		Name:     cfg.Name,
		Subjects: cfg.Subjects,
		Sources: []*jetstream.StreamSource{{
			Name: fmt.Sprintf("OUTBOX_%s", peerSiteID),
			SubjectTransforms: []jetstream.SubjectTransformConfig{{
				Source:      src,
				Destination: dst,
			}},
		}},
	})
	if err != nil {
		return fmt.Errorf("update stream %s: %w", cfg.Name, err)
	}
	return nil
}

// waitForStream polls for stream existence, with a generous timeout to allow
// inbox-worker startup. 30s covers the worst-case container startup; longer
// waits indicate a real provisioning bug worth surfacing.
func waitForStream(ctx context.Context, js jetstream.JetStream, name string) error {
	deadline := time.Now().Add(30 * time.Second)
	for time.Now().Before(deadline) {
		if _, err := js.Stream(ctx, name); err == nil {
			return nil
		}
		select {
		case <-time.After(500 * time.Millisecond):
		case <-ctx.Done():
			return ctx.Err()
		}
	}
	return fmt.Errorf("stream %s did not appear within 30s", name)
}
```

```go
	deadline := time.Now().Add(20 * time.Second)
	for time.Now().Before(deadline) {
		stream, err := js.Stream(ctx, streamName)
		if err == nil {
			cons := stream.ListConsumers(ctx)
			for ci := range cons.Info() {
				if ci.NumWaiting > 0 {
					return
				}
			}
		}
		select {
		case <-time.After(250 * time.Millisecond):
		case <-ctx.Done():
			t.Fatalf("ctx canceled awaiting %s consumer ready", streamName)
		}
	}
	t.Fatalf("no consumer parked on %s within 20s", streamName)
}

// awaitMessage subscribes to subj and returns when the first message arrives,
// failing the test if no message arrives within timeout.
func awaitMessage(t *testing.T, sub *nats.Subscription, timeout time.Duration) *nats.Msg {
	t.Helper()
	msg, err := sub.NextMsg(timeout)
	require.NoError(t, err, "no message on %s within %s", sub.Subject, timeout)
	return msg
}
```

```go
func TestMessage_SendAndBroadcast_SingleSite(t *testing.T) {

	site := e2e.Stack().SiteA

	// Wait for message-worker to be ready before publishing.
	awaitConsumerReady(t, ctx, site.JetStream(t), stream.MessagesCanonical(site.SiteID).Name)

	require.NoError(t, requestReply(alice.Conn(), subject.RoomAddMember(alice.Account, site.SiteID), inviteReq, 5*time.Second, nil))

	// 3. bob subscribes to the broadcast subject.
	bobSub, err := bob.Conn().SubscribeSync(subject.RoomBroadcastWildcard(site.SiteID))

	msg := awaitMessage(t, bobSub, 10*time.Second)
```
