# Pin / Unpin / Favorite demo

`demo-pin-fav.sh` is a thin [`nats` CLI](https://github.com/nats-io/natscli) driver
for the pin/unpin and favorite RPCs against the local-dev stack. It does
**request → reply** only — each action is a **discrete subcommand** that fires one
RPC and prints the raw reply. It does **not** create the room or message; use
`demo-setup.sh` (below) for that.

Two companion scripts live alongside it:
- **`demo-setup.sh`** — one command to make the stack ready and mint a fresh room + message.
- **`demo-state.sh`** — read-only snapshot of the streams / Cassandra rows that move.

## Subcommands

```
demo-pin-fav.sh <action> [args]
```

| Action | Args | Subject | Service |
|--------|------|---------|---------|
| `pin` | `<ROOM_ID> <MESSAGE_ID>` | `chat.user.{acct}.request.room.{room}.{site}.msg.pin` | history-service |
| `unpin` | `<ROOM_ID> <MESSAGE_ID>` | `…msg.unpin` | history-service |
| `list` | `<ROOM_ID>` | `…msg.pinned.list` | history-service |
| `fav` | `<ROOM_ID>` | `…favorite.toggle` → `favorite=true` | room-service |
| `unfav` | `<ROOM_ID>` | `…favorite.toggle` → `favorite=false` | room-service |
| `toggle` | `<ROOM_ID>` | `…favorite.toggle` (single flip) | room-service |
| `all` | `<ROOM_ID> <MESSAGE_ID>` | the whole sequence, in order | both |

`favorite.toggle` is a pure server-side flip (no target state). `fav`/`unfav` make it
**idempotent** by reading the returned `favorite` and toggling once more only if needed
(≤2 RPCs); `toggle` just flips once and reports the new state.

## Prerequisites

1. **The stack containers exist** — run this once so `demo-setup.sh` has something to heal:
   ```bash
   make deps-up   # NATS, Mongo, Cassandra, …
   make up        # auth-service, room-service, history-service, workers, …
   ```
2. **`jq` installed.** (The `nats` CLI is installed automatically by `demo-setup.sh`
   to `/home/codespace/.local/bin` — a persistent path, since `/tmp` gets wiped by the
   dev sandbox. To install it yourself: `GOBIN=/home/codespace/.local/bin go install
   github.com/nats-io/natscli/nats@latest`.)
3. **Creds file** at `docker-local/backend.creds` (generated by `./docker-local/setup.sh`,
   which `make deps-up` runs on first start).

## Quick start

`demo-setup.sh` heals the stack (Vault key, downed services), installs the `nats`
CLI if missing, and **mints a fresh room + message**, printing the ids (also saved
to `/tmp/room_id.txt` / `/tmp/msg_id.txt`):

```bash
./docker-local/demo-setup.sh
export PATH=/home/codespace/.local/bin:$PATH
```

`demo-setup.sh` prints the freshly-minted ids (and saves them to `/tmp/room_id.txt`
/ `/tmp/msg_id.txt`). The current ones are:

```
ROOM_ID = GajUvvpBrQ0H4CS4r
MSG_ID  = TDTxlwffjf2teHcVE3VB
```

Then fire whichever action you want (these run against the ids above):

```bash
./docker-local/demo-pin-fav.sh pin    GajUvvpBrQ0H4CS4r TDTxlwffjf2teHcVE3VB
./docker-local/demo-pin-fav.sh list   GajUvvpBrQ0H4CS4r
./docker-local/demo-pin-fav.sh unpin  GajUvvpBrQ0H4CS4r TDTxlwffjf2teHcVE3VB
./docker-local/demo-pin-fav.sh fav    GajUvvpBrQ0H4CS4r
./docker-local/demo-pin-fav.sh unfav  GajUvvpBrQ0H4CS4r
./docker-local/demo-pin-fav.sh all    GajUvvpBrQ0H4CS4r TDTxlwffjf2teHcVE3VB   # whole sequence
```

> The ids above are a live example from the last `demo-setup.sh` run. Re-running
> setup mints new ones — grab them from its output or `cat /tmp/room_id.txt
> /tmp/msg_id.txt`. (A room is poisoned if Vault loses its key, so always use the
> latest minted ids — see Troubleshooting.)

## Watching it live (multiple terminals)

`demo-logs.sh` follows the app services' logs with per-service prefixes (like
`docker compose logs -f`), so you can watch each request land:

```bash
./docker-local/demo-logs.sh           # pin/fav-relevant services
./docker-local/demo-logs.sh all       # every service
```

A two-terminal workflow:

| Terminal | Command |
|----------|---------|
| **1 — logs** | `./docker-local/demo-setup.sh --logs` (sets up, then streams logs here) |
| **2 — drive** | `./docker-local/demo-pin-fav.sh pin GajUvvpBrQ0H4CS4r TDTxlwffjf2teHcVE3VB`, etc. |

`--logs` runs the full setup and then `exec`s `demo-logs.sh` in the same terminal,
so the setup terminal becomes your live log view.

## Watch the state change

`demo-state.sh` is a read-only companion that snapshots the moving parts — run it
before and after `demo-pin-fav.sh` to see the numbers move. It needs **no `nats` CLI
and no creds**: it reads JetStream over the NATS HTTP monitor port (`:8222`) and
Cassandra via `docker exec`.

```bash
./docker-local/demo-state.sh [ROOM_ID] [MESSAGE_ID]
```

It prints:
1. **JetStream streams** — each stream's `msgs` / `bytes` / `last_seq` (sending a
   message bumps `MESSAGES_*` and `MESSAGES_CANONICAL_*`).
2. **`pinned_messages_by_room`** for the room — a row appears on pin, disappears on unpin.
3. **`messages_by_id` pin flag** for the message — `pinned_at` / `pinned_by` go
   `null` → set on pin, back to `null` on unpin.

Typical loop:
```bash
RID=GajUvvpBrQ0H4CS4r ; MID=TDTxlwffjf2teHcVE3VB
./docker-local/demo-state.sh   $RID $MID         # before — 0 pinned rows, pinned_at=null
./docker-local/demo-pin-fav.sh pin $RID $MID
./docker-local/demo-state.sh   $RID $MID         # after  — note the counts / pin flag
```

Env overrides: `NATS_MON_URL` (default `http://localhost:8222`),
`CASS_CONTAINER` (default `chat-local-cassandra`), `KEYSPACE` (default `chat`).

## Feature reference

What the two features actually enforce server-side (so the replies above make sense).

### Raw NATS requests (no script)

Every action is a NATS **request → reply**. The subject carries `{account}` and
`{roomID}` (natsrouter extracts them); the body is JSON. Fire any of them directly:

```bash
NC="nats --server nats://localhost:4222 --creds docker-local/backend.creds"
$NC req "<subject>" '<body>' --raw
```

| Action | Subject (`{site}` = `site-local`) | Request body |
|--------|-----------------------------------|--------------|
| pin | `chat.user.{account}.request.room.{roomID}.{site}.msg.pin` | `{"messageId":"<id>"}` |
| unpin | `chat.user.{account}.request.room.{roomID}.{site}.msg.unpin` | `{"messageId":"<id>"}` |
| list pinned | `chat.user.{account}.request.room.{roomID}.{site}.msg.pinned.list` | `{}` &nbsp;or&nbsp; `{"cursor":"<c>","limit":50}` |
| favorite | `chat.user.{account}.request.room.{roomID}.{site}.favorite.toggle` | *(empty — no body)* |

Concrete examples (account `alice`, room `GajUvvpBrQ0H4CS4r`, message `TDTxlwffjf2teHcVE3VB`):

```bash
# pin
$NC req "chat.user.alice.request.room.GajUvvpBrQ0H4CS4r.site-local.msg.pin" \
  '{"messageId":"TDTxlwffjf2teHcVE3VB"}' --raw

# unpin
$NC req "chat.user.alice.request.room.GajUvvpBrQ0H4CS4r.site-local.msg.unpin" \
  '{"messageId":"TDTxlwffjf2teHcVE3VB"}' --raw

# list pinned (body REQUIRED — empty body is rejected)
$NC req "chat.user.alice.request.room.GajUvvpBrQ0H4CS4r.site-local.msg.pinned.list" \
  '{}' --raw

# favorite (no body; pure toggle)
$NC req "chat.user.alice.request.room.GajUvvpBrQ0H4CS4r.site-local.favorite.toggle" \
  '' --raw
```

### Pin / Unpin — history-service

Both `pin` and `unpin` share a pre-check: **kill-switch → subscription → message lookup**.

- **Kill switch.** `PIN_ENABLED` (default `true`). When off, every pin/unpin is
  `forbidden` / reason `pin_disabled`.
- **Subscription-gated.** The caller must be subscribed to the room, else `forbidden`
  / reason `not_subscribed` ("not subscribed to room").
- **Message must exist.** `messageId` is required (`bad_request` "messageId is required"
  if empty); unknown id → `not_found` "message not found".
- **Deleted messages.** `pin` rejects a soft-deleted message (`not_found`); `unpin`
  *accepts* it on purpose — a soft-deleted pin still holds a slot and must be removable.
- **Idempotent.** Pinning an already-pinned message echoes the existing `pinnedAt` with
  no write/event; unpinning a non-pinned message is a no-op success.
- **Per-room cap.** `MAX_PINNED_PER_ROOM` (default `10`). At the cap → `forbidden` /
  reason `pin_limit_reached`. (A half-applied "orphan" pin is detected and reuses its
  slot rather than counting twice.)
- **Large-room guard.** In rooms with more than `LARGE_ROOM_THRESHOLD` members
  (default `500`), ordinary members cannot pin/unpin → `forbidden` / reason
  `pin_room_too_large`. **Owners, admins, and bots bypass** this.
- **Events.** A successful pin/unpin best-effort publishes a canonical event
  (`chat.msg.canonical.{site}.pinned` / `.unpinned`) for downstream + cross-site sync.
- **Responses.** `pin` → `{messageId, pinnedAt}` (`pinnedAt` is UTC millis);
  `unpin` → `{messageId}`.

| Error | code | reason |
|-------|------|--------|
| pinning disabled | `forbidden` | `pin_disabled` |
| not a room member | `forbidden` | `not_subscribed` |
| missing `messageId` | `bad_request` | — |
| message unknown / pinning a deleted message | `not_found` | — |
| room already has 10 pins | `forbidden` | `pin_limit_reached` |
| ordinary member in a >500-member room | `forbidden` | `pin_room_too_large` |

### List pinned — history-service

- **Subscription-gated** (same `not_subscribed` reason via the room's history-access check).
- **JSON body required** — send at least `{}`; an empty body is rejected.
- **Cursor pagination.** Request `{cursor?, limit}`; `limit` defaults to `50`, capped at
  `100`. A bad cursor → `bad_request` "invalid pagination cursor". Response is
  `{messages, nextCursor?, hasNext}`, newest-first (`pinned_at DESC, message_id DESC`).
- **History-access redaction.** Pins for messages created *before* the member's
  history-shared-since point are returned as **stub placeholders**: rich content (text,
  mentions, attachments, file, card, quoted parent, reactions, system-message data) is
  blanked while identifiers/sender/timestamps remain, so the client renders a "message
  unavailable" placeholder. Thread replies additionally gate on the parent's timestamp.

### Favorite — room-service

- **Subject** `…favorite.toggle`, **no body** — a pure server-side flip of the
  subscription's `favorite` boolean (Mongo `findOneAndUpdate`, `$not $ifNull false`).
- **Membership-gated.** The caller must have a subscription to the room, else
  `forbidden` / reason `not_room_member` ("only room members can perform this action").
- **Per-member, per-room.** Favorite lives on the caller's own subscription — it never
  affects other members.
- **Events.** Publishes a `favorite_toggled` subscription update; if the caller's home
  site differs from this site, it also emits a cross-site **outbox** event
  (`subscription_favorite_toggled`) so the favorite state federates.
- **Response.** `{status:"ok", favorite:<new bool>}`. There is no target-state input —
  `demo-pin-fav.sh fav`/`unfav` reach a specific state by reading this `favorite` and
  toggling again only if needed.

### Config knobs (history-service)

| Env var | Default | Effect |
|---------|---------|--------|
| `PIN_ENABLED` | `true` | master on/off for pin & unpin |
| `MAX_PINNED_PER_ROOM` | `10` | per-room pin cap |
| `LARGE_ROOM_THRESHOLD` | `500` | member count above which non-privileged users can't pin |

## Troubleshooting

- **`bad_request: invalid request payload` on `pinned.list`** — the handler needs a
  JSON body (`{}`); an empty body fails. The script sends `{}`.
- **`cannot reach NATS` / connection errors** — deps aren't up; run `make deps-up`.
  Note the NATS container's healthcheck can read *unhealthy* until `auth-service` pushes
  the account JWT, even though it's serving — `make up` starting the services resolves it.
- **`pin` returns `internal error` with `Undefined column name pinned_at` (or `reactions`)
  in `pinned_messages_by_room`** — your persistent `cassandra-data` volume has a stale
  schema from an earlier branch. Per `docker-local/cassandra/init/12-table-pinned_messages_by_room.cql`,
  drop the table or wipe the volume and re-init:
  ```bash
  docker compose -f docker-local/compose.deps.yaml down -v   # wipe + re-run make deps-up
  # or just: cqlsh -e 'DROP TABLE chat.pinned_messages_by_room'  then re-apply the init DDL
  ```
- **Room create fails with "must include at least one of users, orgs, channels, or name"**
  — a name alone isn't enough once resolution runs; include a real member
  (`"users":["bob"]`).
- **`internal error` with `cipher: message authentication failed` (or message/history
  services crash-looping on Vault)** — Vault runs in **dev mode (in-memory)**, so a Vault
  restart loses the `chat-kek` transit key. Every room created under the old key is then
  permanently un-decryptable, and the encryption-dependent services die on startup.
  Recover by recreating the key and restarting those services, then use a **fresh room**
  (old rooms stay poisoned):
  ```bash
  docker exec -e VAULT_ADDR=http://127.0.0.1:8200 -e VAULT_TOKEN=dev-only-token \
    chat-local-services-vault-1 sh -c \
    'vault secrets enable -path=transit transit 2>/dev/null; vault write -f transit/keys/chat-kek'
  docker start chat-local-services-{history-service,room-service,message-worker}-1
  # then create a NEW room + message and demo against those IDs
  ```
