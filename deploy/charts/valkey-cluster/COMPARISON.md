# `valkey-cluster` custom chart vs. `bitnami/valkey-cluster` 3.0.x

A side-by-side feature comparison, written so you can sign off (or push back)
on each row before taking this to production.

**Legend:**

- 🟢 Parity with Bitnami
- 🟡 Different by design — does the same thing, possibly more elegantly
- 🔵 Intentionally simpler (Bitnami feature dropped because it's not used here)
- 🟠 Stub / not implemented yet (extension point present, behavior not)
- 🔴 Genuinely missing — flag this before merging if you need it
- ➕ We do something Bitnami doesn't

This document tracks `bitnami/valkey-cluster` chart version **3.0.24** (the one
you pulled, appVersion `8.1.3`) versus this chart at the head of branch
`claude/custom-valkey-cluster-chart`.

---

## TL;DR

| Concern | Verdict |
|---|---|
| Image hygiene (no Bitnami legacy CVEs) | ➕ This chart wins decisively — uses upstream `valkey/valkey:8.1.4-alpine` |
| Day-1 install + bootstrap | 🟢 Parity |
| Day-2 operations (failover, scale-up, pod restart) | 🟢 Parity, verified live |
| Day-2 operations (scale-down) | 🟡 Both require manual reshard — neither chart automates it safely |
| Hardening (NetworkPolicy, securityContext, PDB) | 🟢 Parity; NetworkPolicy is opt-in deny-by-default which is *stricter* than Bitnami's default |
| Observability (Prometheus) | 🟢 Parity; adds 7 ready-made alerts Bitnami doesn't ship |
| Configurability / escape hatches (`extraXxx`) | 🟢 Parity for the common ones |
| TLS in transit | 🟠 Stub only; Bitnami implements it |
| External access (NodePort/LoadBalancer per pod) | 🔵 Intentionally omitted — in-cluster only |
| Sentinel bridge migration mode | 🔵 Omitted — not applicable to greenfield |
| Resource presets (nano/micro/small/...) | 🔵 Omitted — set `resources` directly |
| Lines of YAML (templates/*.yaml) | ➕ ~1,070 vs Bitnami's ~3,860 (≈3.6× smaller) |

If TLS in transit isn't required by your compliance team, this chart is
production-ready. If TLS is required, we either implement it (~150 LOC) or
fall back to Bitnami Secure Images.

## 1. Core deployment

| Feature | Bitnami | This chart | Notes |
|---|---|---|---|
| StatefulSet with N nodes | ✅ | 🟢 | `cluster.masters` × `(1 + replicasPerMaster)` |

## 10. External access

| Feature | Bitnami | This chart | Notes |
|---|---|---|---|
| ClusterIP Service for in-cluster clients | ✅ | 🟢 | Headless Service serves as both |
| NodePort per-pod for external cluster clients | ✅ via `cluster.externalAccess.service.type=NodePort` | 🔵 omitted | Out of scope: this chart assumes in-cluster clients only |

If you ever need external access, use a cluster-aware client running in the
same cluster, or run a `valkey-cli`-as-proxy pod. Exposing Valkey externally
opens a class of authentication / topology-discovery problems we'd rather not
solve in a values file.

## 13. Bitnami-specific features we intentionally omit

These either don't apply to our deployment model or add complexity for no value.
Document them explicitly so you can confirm none of them are blocking
requirements.

| Feature | Why omitted |
|---|---|
| Sentinel bridge mode | We're cluster-mode from day 1; Sentinel doesn't apply |
| HostNetwork mode | Niche to bare-metal latency hunts; hostile to upgrades |
| HostAliases / custom DNS config / scheduler name | Almost nobody uses these |
| Resource presets (`master.resourcesPreset`) | Marketing; set `resources` directly |
| `cluster.externalAccess.*` | We're in-cluster only |
| `tls` (full implementation) | Stub only — see row in §2 |
| `pdb.minAvailable` (we use `maxUnavailable: 1` by default) | Equivalent — `maxUnavailable: 1` is more drain-friendly |
| `master.disableCommands` (rename/disable specific Valkey commands) | Add via `config.extra` if you need it |
| `master.priorityClassName` (separate priority class for master vs replica) | Cluster topology is fluid (replicas get promoted) — single priority class is correct |
| `metrics.podMonitor` | We ship `ServiceMonitor` only |
| `metrics.extraArgs` for exporter | Add via `extraEnvVars` on the metrics container (would need a values addition; flag if needed) |

## Confidence assessment for production

**Go to prod if:**

- ✅ Your security team does NOT require TLS in transit for in-cluster Valkey traffic.
- ✅ Your clients are in-cluster (no external NodePort/LB needed).
- ✅ You're OK setting `networkPolicy.allowedClients` to declare who's allowed to connect (and your CNI implements NetworkPolicy — Calico, Cilium, etc.).
- ✅ You can run the dev-cluster smoke commands in `runbooks/test-report.md` to verify the K8s-only pieces (NetworkPolicy enforcement, helm hooks, StatefulSet rolling update) against your actual cluster.

**Hold and address first if:**

- ⚠️ Compliance requires encryption-in-transit. Fix: implement the TLS stub (~150 LOC) or fall back to Bitnami Secure Images.
- ⚠️ You need external (cross-cluster, off-cluster) client access. Fix: in-cluster client proxy, or extend the chart with per-pod NodePort services.
- ⚠️ You want `existingConfigmap` (full external valkey.conf). Fix: ~10 LOC addition to `statefulset.yaml`.

Anything else from the comparison rows — raise it before production cutover and
we'll close the gap.

## How to compare the rendered output yourself

If you want byte-level diffing between what this chart produces and what
Bitnami's chart produces for an equivalent topology:

```bash
# Bitnami (assuming you've vendored 3.0.24 locally)
helm template valkey ./bitnami-valkey-cluster-3.0.24/ \
  --set cluster.nodes=6 \
  --set cluster.replicas=1 \
  --set auth.existingSecret=valkey-auth \
  > /tmp/bitnami.yaml

# Ours
helm template valkey ./deploy/charts/valkey-cluster/ \
  -f ./deploy/charts/valkey-cluster/values-production.yaml \
  > /tmp/ours.yaml

# Diff by resource kind
diff <(grep '^kind:' /tmp/bitnami.yaml | sort -u) \
     <(grep '^kind:' /tmp/ours.yaml | sort -u)
```

The intentional differences will be: NodePort/LB Services in Bitnami's output
that we don't render, and our `PrometheusRule` + `helm test Pod` that Bitnami
doesn't render. Everything else should be functionally equivalent.
