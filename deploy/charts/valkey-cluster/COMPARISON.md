## Confidence assessment for production

**Go to prod if:**

- тЬЕ Your security team does NOT require TLS in transit for in-cluster Valkey traffic.
- тЬЕ Your clients are in-cluster (no external NodePort/LB needed).
- тЬЕ You're OK setting `networkPolicy.allowedClients` to declare who's allowed to connect (and your CNI implements NetworkPolicy тАФ Calico, Cilium, etc.).
- тЬЕ You can run the dev-cluster smoke commands in `runbooks/test-report.md` to verify the K8s-only pieces (NetworkPolicy enforcement, helm hooks, StatefulSet rolling update) against your actual cluster.

**Hold and address first if:**

- тЪая╕П Compliance requires encryption-in-transit. Fix: implement the TLS stub (~150 LOC) or fall back to Bitnami Secure Images.
- тЪая╕П You need external (cross-cluster, off-cluster) client access. Fix: in-cluster client proxy, or extend the chart with per-pod NodePort services.
- тЪая╕П You want `existingConfigmap` (full external valkey.conf). Fix: ~10 LOC addition to `statefulset.yaml`.

Anything else from the comparison rows тАФ raise it before production cutover and
we'll close the gap.

## How to compare the rendered output yourself

If you want byte-level diffing between what this chart produces and what
Bitnami's chart produces for an equivalent topology:

```bash
# Bitnami (assuming you've vendored 3.0.24 locally)
helm template valkey ./bitnami-valkey-cluster-3.0.24/ \
  --set cluster.nodes=6 \
  --set cluster.replicas=1 \
  --set auth.existingSecret=valkey-auth \
  > /tmp/bitnami.yaml

# Ours
helm template valkey ./deploy/charts/valkey-cluster/ \
  -f ./deploy/charts/valkey-cluster/values-production.yaml \
  > /tmp/ours.yaml

# Diff by resource kind
diff <(grep '^kind:' /tmp/bitnami.yaml | sort -u) \
     <(grep '^kind:' /tmp/ours.yaml | sort -u)
```

The intentional differences will be: NodePort/LB Services in Bitnami's output
that we don't render, and our `PrometheusRule` + `helm test Pod` that Bitnami
doesn't render. Everything else should be functionally equivalent.
